# 竞品情报看板使用与业务口径说明

> 本文是看板业务口径的统一说明，面向运营、产品、市场及管理人员。阅读本文不需要数据、AI 或开发背景。它重点解释每个数字、标签和结论代表什么，应该如何使用，以及哪些结论仍需人工确认。

## Executive Summary

- **看板记录的是“本次采集发现了什么”**。顶部“最新批次”只展示最近一次成功批次发现的新增或更新内容；“新增”不一定等于竞品当天发布，仍需同时查看公告发布日期。
- **Campaign 和 Product 会与 Zoomex 历史能力比较**，差异统一为四种展示结果：未检索到同类、已有同类但机制不同、已有同类型但仅能做粗粒度参考、未进行对比。已完成分析的 Campaign/Product 不应出现“未进行对比”。
- **优先级有两套，作用不同**。高/中/低用于判断单条 Campaign/Product 的关注程度；P1～P7只决定 Overview Highlights 的阅读顺序，二者互不替代。
- **Listing & Delisting 只做事实整理和币种赛道分类**，不做 Zoomex 对比、AI 内容总结或高/中/低优先级判断。

## 0. 第一次使用时，建议这样看

每天打开看板后，按以下顺序即可完成一次基础情报检查：

1. 点击顶部 **“最新批次”**，确保没有遗留的自定义日期范围。
2. 在 Overview 查看四类数量和 Competitive Insight，先判断今天的变化集中在哪个平台、类别和市场。
3. 阅读 Highlights：P1～P5通常比普通上币和一般公告更值得先看。
4. 进入 Campaign 或 Product，优先检查“未检索到同类”“机制不同”和高优先级内容。
5. 展开 View Detail，核对竞品原文、Zoomex 对照证据以及差异理由。
6. 进入 Listing & Delisting 查看币种赛道、现货/合约类型和平台分布；这部分由业务人员自行判断是否值得跟进。
7. 如果需要研究某个市场，在 Markets 查看跨地区覆盖；如果需要查历史，在 Search 使用近30/90天、近1年或顶部日期范围。

任何需要立项、预算、产品排期或合规判断的结论，都应回到源公告复核。看板负责缩小阅读范围，不负责代替最终决策。

## 1. 这个看板解决什么问题

看板持续跟踪五家竞品交易所的公开公告：

- Bitunix
- Weex
- BingX
- Phemex
- Lbank

Zoomex（下文简称 **ZMX**）不作为竞品统计，而是作为我方对比基线。

看板希望帮助业务人员快速回答以下问题：

1. 竞品最近推出了哪些新活动、新产品或上币动作？
2. 哪些公告只是普通通知，哪些值得优先关注？
3. 同一事件覆盖了哪些国家或语言市场？是否仅在特定地区出现？
4. ZMX 是否已经有类似活动或产品？
5. 如果 ZMX 已有类似能力，竞品在玩法、奖励、门槛或目标用户上有什么不同？
6. 哪些现象只是单一竞品尝试，哪些可能已经成为行业趋势？

看板的作用是帮助筛选和定位信息，不代替最终的业务决策。涉及产品立项、预算投入或合规判断时，仍应打开原公告并由对应负责人复核。

## 2. 从公告到看板结论的整体过程

整个过程可以理解为六步：

```text
采集公开公告
    ↓
识别新增、变更和重复内容
    ↓
判断公告属于活动、产品、上币、下架还是普通公告
    ↓
提取玩法、奖励、时间、目标用户等关键信息
    ↓
与 ZMX 历史能力和具体案例比较，形成看板结论
    ↓
生成三份整体 Insight，并同步看板、飞书三表和 EN 群 Overview 截图
```

常规每日流程只处理指定 UTC 日期内发现的新增或更新竞品数据，不会每次重新抓取全部竞品历史。Zoomex 的历史能力目录会作为长期对比基线维护；历史补跑是单独的维护动作，不属于每天默认采集。

### 第一步：收集公告

系统从各交易所公开的 **Announcement 公告中心**获取内容；对于已接入专门活动入口的竞品，还会同时采集其活动中心、奖励页等活动板块。两类入口共同构成看板的竞品内容来源，并覆盖各自已配置的语言市场：

| 平台 | 当前覆盖市场 |
|---|---|
| Bitunix | EN、FR、ID |
| Weex | EN、FR |
| BingX | EN、VN |
| Phemex | EN、FR |
| Lbank | EN、VN、ID |
| ZMX | EN、FR、EN-Asia、VN、ID |

没有配置的数据源或语言不会被估算。例如当前没有 KR 采集源，所以看板不会根据其他市场推测韩国市场情况。

### 第二步：识别新增和变化

系统会保存每篇公告的内容特征，并与之前采集到的版本比较：

- **新增（new）**：系统第一次看到这篇公告；
- **变更（changed）**：之前已经看到过，但正文发生变化；
- **未变（unchanged）**：内容与上次相同。

需要注意：**新增表示系统第一次采集到，不一定表示公告今天刚发布。**

例如第一次补录一篇两个月前的历史公告时，它在数据层仍然是 new。为避免误导，Overview 的重点排序会把发布时间早于本批次 30 天以上的补录内容降到最低层级，不把它包装成“今日新活动”。

### 第三步：分类

不同类别的数据来源口径如下：

- **Campaign**：专门活动入口中的活动，与 Announcement 中识别为活动的内容取并集；合并后按业务事件及跨语言版本去重。
- **Product**：从 Announcement 内容中识别出的产品能力、功能或规则变化。
- **Listing & Delisting**：从 Announcement 内容中识别出的上币与下架事实。
- **Announcement（其他）**：Announcement 中排除 Campaign、Product、Listing 和 Delisting 后的其余公告，例如维护和一般通知。

因此，Campaign 并不只来自公告中心，也不只来自活动中心；Product、Listing、Delisting 和其他公告则以 Announcement 的分类识别结果为来源。

每篇公告会被归入以下类别之一：

| 类别 | 业务含义 | 典型内容 |
|---|---|---|
| Campaign | 营销或用户运营活动 | 交易赛、充值奖励、空投、邀请活动、抽奖 |
| Product | 产品能力或规则更新 | 新功能、产品上线、费率/风控/UI 调整 |
| Listing | 上币 | 现货或合约交易对上线 |
| Delisting | 下架 | 交易对、代币或相关服务停止 |
| Announcement | 其他公告 | 维护、一般通知或暂不属于以上类别的内容 |

分类会参考源站原始栏目、标题和正文关键词。它是为了方便业务阅读而做的统一口径，不等同于竞品网站自己的栏目命名。

### 第四步：合并重复和多语言版本

同一业务事件可能同时发布 EN、FR、VN 等多个版本。如果直接计数，会把一件事算成多件事。

看板按以下顺序合并：

1. 系统已经确认属于同一跨语言事件；或
2. 同一平台、不同语言、标题规范化后完全相同。

合并后：

- 看板主表只展示一行；
- `markets` 显示该事件覆盖的所有语言市场；
- 详情中仍保留每种语言的原文链接；
- Summary 和 Highlights 不会因为多语言版本重复计数。

### 第五步：分析并与 ZMX 比较

系统只对 **Campaign 和 Product** 做深入分析与 ZMX 对比。

Listing 和 Delisting 使用同一个轻量分析流程：LLM 只判断币种所属赛道，不生成 ZMX 差异、分析优先级或跟进建议。Spot/Perpetual、Token、交易对和上下架状态仍由公告事实和固定规则识别。

## 3. 看板中的“今天”是什么

看板中的“最新批次”不是简单取用户打开页面的自然日期，也不是取服务器当前时间。

系统会找到数据库中最近一次成功采集到竞品数据的日期，并把它作为看板批次日期。

这样处理的原因是：

- 数据可能在第二天补跑；
- 可能进行历史补数；
- 服务器和业务人员所在时区可能不同；
- 页面发布可能晚于实际采集时间。

因此，“今日数据”更准确的理解是：**最近一次采集批次发现的数据**。

页面顶部选择“最新批次”时，展示这一批次中状态为新增或变化的内容；选择自定义日期后，页面会切换到全历史数据再进行筛选。

顶部的平台、市场和日期条件是全局条件，会同步作用于 Overview、Campaign、Product、Listing & Delisting、Markets 和 Search。点击“最新批次”会退出自定义日期范围并恢复最近一次采集批次；平台和市场选择会保留，方便继续查看同一对象。

### 3.1 顶部全局筛选器

| 控件 | 作用 | 需要注意 |
|---|---|---|
| From / To | 按公告发布日期选择范围 | 它不是按分析生成时间筛选；只填写一端时表示开放区间 |
| 最新批次 | 清除 From/To，恢复最近一次成功批次 | 不会清除已经选择的平台和市场 |
| Exchange | 只看某一家竞品 | Zoomex 是对比基线，不在竞品选择项中 |
| Market | 只看覆盖某语言市场的事件 | Markets 中 VN、ID会映射到 SEA 视角，但底层仍分别保留 VN 与 ID |

各业务 Tab 内还可以继续使用局部筛选器。全局和局部条件是“同时满足”的关系，例如顶部选择 Weex、Campaign 内再选择交易竞赛，结果只包含 Weex 的交易竞赛。

### 3.2 时间基准

看板时间统一按 UTC 展示。北京时间比 UTC 快8小时，因此北京时间凌晨看到的“今天”，可能仍属于上一个 UTC 日期。页面顶部的日期和群截图批次均以 UTC 为准。

## 4. Overview 首页如何理解

Overview 用于快速判断本批次发生了什么，主要包括 Summary、Highlights、Today’s Insight 和 Trend。

### 4.1 四个 Summary 数字

首页四个数字分别表示：

- Campaign：新增或变化的活动事件数；
- Product：新增或变化的产品事件数；
- Listing & Delisting：上币与下架事件总数；
- Announcement：其他公告事件数。

计数已经按业务事件去重。同一活动的 EN 和 FR 版本不会被算成两条。

Campaign 和 Product 还会显示 ZMX 差异分布，例如：

- 未检索到同类；
- 已有同类，但机制不同；
- 已有同类型，但只能做粗粒度参考；
- 未进行对比。

这四项互斥，同一条业务事件在同一个页面范围内只会计入其中一项。Listing & Delisting 和 Announcement 不展示这组差异数字。

当用户选择平台、市场或日期筛选后，这些数字会根据筛选结果重新计算。

### 4.2 Highlights 为什么是这几条

Highlights 不是简单按发布时间取最新五条，而是按照业务价值排序：

| 顺序 | 内容 |
|---|---|
| P1 | 新活动 |
| P2 | 活动规则变化 |
| P3 | 活动奖励变化 |
| P4 | 新产品 |
| P5 | 产品更新 |
| P6 | 上币或下架 |
| P7 | 其他公告及较早的历史补录 |

同一层级内再按发布时间从新到旧排序。

为了避免某一家竞品刷屏，默认 Top 5 中每个平台最多出现 2 条。

这里的 P1～P7 只用于首页信息排序，不代表业务分析中的“高/中/低优先级”。两者不能混为一谈。

P1～P7不读取 LLM 给出的高/中/低，也不读取差异标签。每个平台最多进入2条，是为了避免单个平台占满 Highlights；因此 Highlights 不是所有高优先级内容的完整清单，需要完整排查时应进入 Campaign、Product 或 Search。

### 4.3 Today’s Insight 从哪里来

Today’s Insight 有两种来源：

1. **LLM 生成**：常规每日流程会汇总全部已覆盖市场的批次分析，生成并缓存 2～4 句综合分析，重点说明活动、产品、市场和整体变化特点；
2. **范围实时汇总**：用户调整日期、平台或市场后，页面根据当前筛选结果即时生成同样结构的 2～4 句中性概括。

三张领先平台卡使用确定性统计，分别展示新增活动、新增产品和奖励变化最多的平台，不由模型猜测。打开页面或导出数据时不会临时调用模型，所以页面内容不会因为刷新而随机变化。

只有未使用全局自定义筛选、且已命中当前批次 LLM 缓存时，正文才标记为“AI 自动总结”；其他范围标记为“范围实时汇总”。若当前范围没有明显变化，统一显示 `No Significant Changes Today`。

### 4.4 Trend 表达什么

Trend 展示系统在每天的采集过程中发现了多少新增或变化事件，并按 Campaign、Product、Listing 等类别分线显示。

它按“系统发现日期”统计，不是严格按“公告发布日期”统计。因此：

- 某天补录历史公告，可能让当天的发现量升高；
- 它适合观察情报流入和竞品动作密度；
- 不适合直接当作竞品官方发文频率的精确统计。

### 4.5 Overview 图表和悬浮图例怎么读

Platform × Category 使用堆叠条形图：每一行是一家竞品，颜色代表 Campaign、Product、Listing & Delisting。条形总长度表示当前筛选范围内的事件总数，颜色区段表示各类别构成。

图例采用悬浮交互：

- 悬浮某个颜色图例时，重点查看该类别在不同平台中的分布；
- 悬浮某个条形区段时，会显示该平台、该类别和对应数量；
- 图中的数字只是当前筛选范围的去重事件数，不是金额、用户数或业务效果。

顶部平台、市场和日期改变后，图表会同步重新计算。图表用于比较“谁更活跃、动作集中在哪类”，不能据此判断活动效果好坏。

## 5. Campaign 页面规则

Campaign 页面关注活动本身的玩法、奖励、时间、目标人群以及与 ZMX 的差异。

看板中的“差异”统一收敛为四类：未检索到同类、已有同类但机制不同、已有同类型但仅为粗粒度参考、未进行对比。Overview 统计、Campaign/Product 筛选与标签、Markets 展开详情、Search 筛选和飞书业务表均使用同一套口径；历史上的“混合”“已有同类”等旧标签不再作为独立前端分类展示。

Campaign 和 Product 的每条记录都会进入对比流程，因此已分析记录不会显示“未进行对比”。没有召回候选、没有匹配结果或模型判断不适用时，统一归为“未检索到同类”；“未进行对比”只用于确实尚未被分析流程覆盖的历史记录。

### 5.1 页面展示哪些信息

每条活动通常包含：

- 竞品和活动标题；
- 新增或变化状态；
- 覆盖市场；
- 活动机制；
- 奖励金额或奖励形式；
- 目标用户；
- 开始和结束时间；
- ZMX 差异结论；
- 分析优先级；
- 原文链接和其他语言版本。

源公告没有明确提供的字段不会被强行补全。例如没有结束时间时，看板不会猜测活动何时结束。

页面顶部四张卡分别表示：

| 指标 | 统计口径 |
|---|---|
| New Campaign | 当前范围内采集状态为 new 的活动数 |
| Updated Campaign | 当前范围内正文发生变化的活动数 |
| Ongoing Campaign | 有时间信息且当前未开始、未结束条件均不成立的活动数 |
| Ending Soon · ≤3 days | 当前仍在进行且结束日期距批次日期不超过3天的活动数 |

Ending Soon 是提醒指标，不是活动状态筛选项；这些活动在状态上仍属于 Ongoing。

### 5.2 活动类型如何识别

系统会从标题和分析出的活动机制中识别常见玩法，例如：

- 邀请/推荐；
- 交易竞赛；
- 入金激励；
- 空投或 Launch；
- 预测活动；
- 积分活动；
- 抽奖、盲盒、转盘；
- 新用户欢迎礼；
- 手续费返还；
- VIP 专属奖励等。

如果无法可靠归入已有类型，会保留为其他类型，而不是创造一个看似精确的新类别。

### 5.3 奖励信息如何得到

奖励优先来自正文的结构化分析，包括金额、币种和奖励形式。

如果结构化奖励为空，但活动机制描述中存在明确金额，例如 `50 USDT`，展示层会摘取最多三个明确金额作为补充。该规则只摘录文本中已经存在的金额，不自行换算或估算奖励价值。

### 5.4 活动状态如何判断

页面会结合采集状态、公告发布日期以及已经提取到的开始/结束时间判断：

- New：本批次首次发现，且公告发布日期就是当前批次日期；
- Updated：已存在公告的正文发生变化；
- Upcoming：尚未开始；
- Ongoing：已有开始或结束时间，且未满足 Upcoming/Ended；
- Ended：已经结束；
- Date unknown：没有足够时间信息。

“即将结束”目前不是独立状态。若时间信息不完整，状态可能显示 Date unknown，此时应以源公告为准。

### 5.5 活动变化如何识别

当同一公告正文发生变化时，系统会比较新旧内容，重点识别：

- 奖励变化；
- 规则变化；
- 延期；
- 结束或取消；
- 其他更新。

首页会优先突出规则和奖励变化，因为它们通常比普通文案修改更有业务意义。

### 5.6 Campaign Insight

Campaign 页面提供 2～4 句整体总结，覆盖当前筛选范围内的活动类型、重点市场、主要玩法、奖励趋势和整体特点。默认最新批次优先使用 LLM 专项总结；选择平台、类型、状态、市场、差异或关键词后，改为根据筛选后的活动实时汇总，确保总结与下方列表一致。

### 5.7 Campaign 默认排序

Campaign 列表不是按照高/中/低优先级排序，而是采用稳定的业务顺序：

1. 当天发布且本批次首次发现的新活动；
2. 已有活动的内容更新；
3. 正在进行的活动；
4. 即将开始的活动；
5. 已结束的活动；
6. 同一组内再按公告发布时间从新到旧排列。

这样可以保证每次刷新后的阅读顺序稳定。高/中/低只作为判断依据展示，业务人员可结合差异标签进一步筛选。

## 6. Product 页面规则

Product 页面关注新产品、新功能以及已有产品的规则或能力变化。

页面顶部四张卡分别表示：

| 指标 | 统计口径 |
|---|---|
| New Product | 被识别为 New Product 的事件数 |
| Updated Product | 除 New Product 外的所有产品更新事件数 |
| Affected Exchange | 当前范围内出现产品变化的不同交易所数量 |
| Affected Category | 当前范围内涉及的不同产品能力类别数量 |

四张卡都会跟随顶部全局条件。页面内继续选择产品类别、状态、市场、差异或关键词后，列表和 Product Insight 会同步变化；顶部卡片保持该 Tab 的全局范围总览，便于先看总体、再看局部。

### 6.1 产品类别

系统根据标题和功能描述识别：

- Copy Trading
- Card & Payment
- Security
- Earn
- API
- Bot
- Wallet
- Convert
- Institutional
- Trading
- Others

这是为了让不同竞品的产品公告使用统一分类口径。无法明确识别时进入 Others。

### 6.2 新产品与产品更新

页面将内容分为两组：

- **New Product**：标题和状态更像新能力上线；
- **Product Update**：已有功能的调整、升级、移除、维护或规则变化。

具体更新类型包括：

- New Product；
- Feature Added；
- Feature Removed；
- UI Updated；
- Performance Updated；
- Rule Updated；
- Operational Notice。

判断主要依据标题中的上线、更新、调整、移除、性能、UI、维护等明确词语，以及系统是否首次采集到该公告。

由于“首次采集”不必然等于“新产品”，标题中明确出现 update、upgrade、adjust、maintenance 等词时，会归为产品更新，不会只因为状态是 new 就标记为新产品。

### 6.3 产品核心能力与内容总结

Product 页面提供 2～4 句整体总结，说明当前范围内主要产品能力、内容重点、重点市场、新产品与既有能力更新的结构以及整体变化特点。默认最新批次优先使用 LLM 专项总结；应用页面筛选后切换为基于当前结果的实时汇总。

### 6.4 Product 默认排序

Product 按业务变化类型排序：New Product、Feature Added、Feature Removed、Rule Updated、UI Updated、Performance Updated、Operational Notice，无法归类的排在最后。同一类型内再按发布时间从新到旧排列。

该顺序强调“新能力和实质变化”，不等于高/中/低评分，也不表示排在前面的产品一定更适合 Zoomex。

## 7. Listing & Delisting 页面规则

该页面合并展示上币和下架事件，并明确标识类型。

页面的 Overview 包含四张指标卡：

- New Spot Listing；
- New Futures Listing；
- Delisting；
- Affected Exchange。

下方同时展示 Category Distribution 和 Exchange Distribution，让运营快速看到当天最集中的币种赛道和最活跃的平台。

当前实际采用两张等高的堆叠图：

- **Category × Exchange**：纵向类别是币种赛道，每种颜色表示一家交易所；
- **Listing Type × Exchange**：纵向类别是 Spot、Perpetual、Spot & Perpetual，每种颜色表示一家交易所。

悬浮图例会高亮对应交易所，悬浮条形区段会显示“类别、交易所、数量”。条形右侧总数表示该类别在当前筛选范围内的事件数。Pending 不进入币种 Category 分布，避免把尚未完成分类的数据伪装成一个业务赛道。

币种 Category 使用固定分类：

- AI；
- Meme；
- Layer2；
- DeFi；
- GameFi；
- RWA；
- DePIN；
- Other。

LLM 只允许从以上类别中选择，不负责判断 ZMX 是否应该上币，也不生成业务优先级。证据不足或属于其他赛道时进入 Other；尚未完成分类时显示 Pending。

Listing Detail 展示：

- 平台；
- 标题；
- 发布时间；
- 市场；
- 上币或下架；
- 现货或合约类型（仅在标题证据明确时）；
- 币种 Category；
- Token 和 Trading Pair；
- Launch Time；
- 原文链接。

现货/合约识别规则为：

- 标题只出现 perpetual、perp、futures、contract 等明确词语时，标为合约；
- 标题只出现 spot 时，标为现货；
- 两类证据同时出现时，标为 Spot & Perpetual，并同时计入 Spot 与 Futures 指标；
- 标题未写类型时，可使用已经核实的来源栏目作为次级证据，例如 Weex New Spot Listings；
- 没有标题、正文或来源栏目证据时才保留未知，不凭空推测。

页面支持按 Exchange、Listing Type、Status、Category、Language 和关键词筛选，默认按发布时间从新到旧排序。主表直接增加 Markets 列，与 Campaign、Product 使用相同的区域表达方式；点击 View Detail 可查看 Trading Pair、完整公告、来源链接等信息。Listing 页面不再单独设置 Region Coverage 表，避免区域信息在主表和独立表中重复展示。

四张指标卡的精确定义：

| 指标 | 统计口径 |
|---|---|
| New Spot Listing | Status 为 New Listing，且 Listing Type 包含 Spot 的事件数 |
| New Futures Listing | Status 为 New Listing，且 Listing Type 包含 Perpetual 的事件数 |
| Delisting | Status 为 Delisted 的事件数，不区分 Spot 或 Perpetual |
| Affected Exchange | 当前范围内至少出现一条 Listing/Delisting 的不同交易所数量 |

一条同时包含 Spot 和 Perpetual 的公告会分别计入前两张卡，因此两张卡相加不一定等于上币公告总数。

该页面不显示历史版本遗留的 ZMX 差异、优先级或跟进建议，避免把已经不符合当前分析政策的旧结论继续呈现给业务人员。“哪些 Listing 最值得关注”和“ZMX 是否需要关注这些币种”属于二期 Listing Insight，目前先通过 Category 与平台分布支持人工判断。

## 8. Markets 页面规则

Markets 页面用于从地区或语言市场观察竞品动作。页面仅展示已有采集覆盖的 EN、FR 和 SEA（VN+ID），不为尚未接入的数据源保留 KR 空入口。

顶部 Campaign、Product、Listing（含下架）三张指标卡只回答“当前市场和筛选范围内有多少条”，不再混放公告链接和 ZMX 对比结论。下方“跨地区公告对比”表支持按 Campaign、Product、Listing & Delisting、其他筛选，每条记录按跨语言 group 合并展示覆盖市场。Campaign 与 Product 点击 View Detail 后可查看 AI Summary、ZMX Comparison 和来源链接；ZMX 对比采用与其他详情字段一致的文字格式，不使用独立 callout。Listing & Delisting 只展示 Token、Listing Type、Status、Category、Markets、日期和来源，不展示 AI 分析或 ZMX 对比。这样指标区用于快速读数，明细区用于追溯和判断。

用户选择 EN、FR 或 SEA 后，页面会筛选所有覆盖该市场的业务事件，并分别展示活动、产品和上币重点。

顶部全局筛选会同时作用于三张业务卡和底部跨地区公告表。底部表不是固定的全历史快照，而是根据当前日期、平台和市场条件重新归组，因此表内数量和覆盖点会随筛选条件变化。

一个事件可能同时属于多个市场。例如同一活动有 EN 和 FR 两个版本，则它会同时出现在 EN 和 FR 视图中，但在全局统计中仍只算一个业务事件。

### 地区独占如何理解

如果同一组业务事件只在部分市场出现，系统可将其标记为地区独占。这通常意味着：

- 区域化运营活动；
- 特定语言市场奖励；
- 当地用户专属门槛；
- 各地区公告发布节奏不同。

“地区独占”首先是发布覆盖事实，不一定代表产品能力真的只对该地区开放。涉及权限或资格时仍需查看正文。

## 9. Search 页面规则

Search 用于从全历史数据中检索公告，可按以下条件组合筛选：

- 平台；
- 市场；
- 类别；
- ZMX 差异；
- 最近 30、90、365 天；
- 标题关键词。

Search 页面不再重复提供具体起止日期输入。需要指定日期区间时使用页面顶部的全局日期筛选；Search 内的地区、时间、来源、分类和差异均支持同行多选，同一行按“任一匹配”、不同行按“同时满足”组合；近 30 天、近 90 天和近 1 年按钮作为快捷时间窗口，并与顶部条件共同生效。

每页展示 20 条，并可将当前筛选结果下载为 CSV。

Search 只提供摘要字段，不保存或下载完整正文。需要核实细节时应点击原文链接。

## 10. ZMX 对比到底是怎么做的

这是看板最核心、也最容易被误解的部分。

ZMX 对比不是简单看标题是否相似，也不是只搜索 ZMX 最近几个月的公告。它分为“建立我方能力目录”和“逐条比较竞品事件”两个阶段。

### 10.1 先建立 ZMX 能力目录

系统读取 ZMX 已采集的全部历史 Campaign 和 Product 公告，提取：

- 活动或产品属于哪种机制；
- 核心玩法或功能；
- 奖励形式和范围；
- 参与门槛；
- 目标用户；
- 开始与结束时间；
- 可作为证据的具体公告。

这些内容会归入预先维护的标准分类中。例如活动中的入金奖励、交易量阶梯、抽奖、预测活动、新用户欢迎礼等；产品中的跟单、理财、钱包、API、交易能力等。

使用标准分类的目的是避免同一种玩法被写成多个近义标签，导致系统误以为它们是不同能力。

目录覆盖 ZMX 全量已采集历史，而不是只看最近 90 天。这样才有条件判断“ZMX 是否曾经有过该能力”。

### 10.2 从目录中找最相关的 ZMX 候选

对于每条竞品公告，系统先提取它的活动机制或产品功能、奖励、资格和目标用户，再从同类别、同市场的 ZMX 目录中寻找最相关案例。

候选召回会比较这些结构化信息是否有共同词项，而不是只比较标题。完全没有关联的 ZMX 案例不会送入下一步，避免模型为了完成任务而硬找一个无关案例。

### 10.3 四种差异标签如何判断

系统结合逐条对比结果和 Zoomex 全量能力目录，统一输出四种业务标签：

| 前端标签 | 判断含义 | 业务应该怎么用 |
|---|---|---|
| 未检索到同类 | 没有找到可支持“Zoomex已有同类”的可靠案例；也包括候选召回为空或分析判断不适用的情况 | 作为待优先人工检查的机会线索，不能仅凭标签直接宣布 Zoomex 一定缺失 |
| 已有同类 · 机制不同 | 找到了可比的 Zoomex 具体案例，但奖励、门槛、目标用户、流程或产品规则存在实质差异 | 展开详情判断差异是否值得借鉴，不要只比较活动名称或产品包装 |
| 已有同类型 · 粗粒度 | Zoomex 全量目录确认做过这一机制类型，但当前竞品公告没有匹配到足够精确的一对一案例 | 只能说明“大类做过”，不能说明具体玩法、奖励或能力完全一致 |
| 未进行对比 | 该条尚未被完整分析流程覆盖 | 不得解释为“没有差异”或“Zoomex已有” |

Campaign 和 Product 的每条新增或更新内容都会进入对比流程。因此，对已经显示分析结果的 Campaign/Product，如果没有匹配到同类，应显示“未检索到同类”，而不是“未进行对比”。“未进行对比”只允许出现在真正尚未分析的历史遗留记录中。

Listing & Delisting 从设计上不做 Zoomex 对比，在它自己的页面和 Markets 详情中不展示这组标签。

### 10.4 对比详情展示什么

展开详情后，页面按两栏展示：

| 竞品侧 | ZMX 侧 |
|---|---|
| 竞品标题 | 被引用的 ZMX 对照公告 |
| 活动机制或产品功能 | ZMX 能力描述 |
| 奖励、目标用户、时间等 | ZMX 对应摘要、奖励及原文链接 |

下方展示差异理由，说明为什么被判断为已有、不同或缺失。

如果没有具体 ZMX 对照公告，只能说明当前没有可展示的一对一案例。此时应区分“粗粒度类型命中”和“未检索到同类”，并结合差异理由及源公告复核。

## 11. “已分析”“待分析”和“未进行对比”的区别

这三个概念需要严格区分：

- **已分析（analyzed）**：该公告已经完成事实提取和 Zoomex 比较流程；最终一定会落入前三种有效差异标签之一；
- **待分析（pending）**：公告存在，但尚未进入完整分析流程；
- **未进行对比**：待分析记录在前端的中性展示，不能解释为“双方没有差异”。

因此，看到空白、未进行对比或待分析时，不应将其纳入“ZMX 已覆盖率”统计。

## 12. 两套“优先级”必须分开理解

### 12.1 单条分析优先级：高 / 中 / 低

每条 Campaign 或 Product 的关注优先级由固定公式计算。LLM 只负责从原文中识别有限的判断要素，最终加分和分档由程序完成，因此同样输入会得到同样结果。

| 判断因素 | 当前分值 |
|---|---|
| 事件类型 | 奖励变化 +30；规则变化 +28；延期 +20；结束 +18；新增或取消 +15；其他更新 +10；无法判断 +0 |
| Zoomex 差异 | 确认缺失 +30；已有同类但机制不同 +22；未找到候选 +12；已有同类 +3；不适用 +0 |
| 业务影响 | 高 +20；中 +12；低 +4 |
| 原文与抽取置信度 | ≥90% 加10；70%～90% 加5；低于70% 扣10 |
| 新颖度 | 0～3级，每级加2分，最多6分 |
| 紧迫度 | 0～3级，每级加2分，最多6分 |

最终分档：

- **高**：70分及以上；
- **中**：40～69分；
- **低**：40分以下。

例如一条“奖励变化”的活动，如果属于“已有同类但机制不同”、业务影响为中、原文置信度高、新颖度和紧迫度各2级，总分是 `30 + 22 + 12 + 10 + 4 + 4 = 82`，因此显示为高。

需要注意：前端只展示高/中/低，不展示中间分数；Listing & Delisting 不参与这套计算。Campaign/Product 默认列表也不会按这个分数排序，避免模型判断改变稳定阅读顺序。

### 12.2 Overview 排序优先级：P1～P7

P1～P7是首页 Highlights 的固定内容排序，不是单条分析得分：

| 层级 | 内容 |
|---|---|
| P1 | 新活动 |
| P2 | 活动规则变化 |
| P3 | 活动奖励变化，以及无法进一步细分的其他活动更新 |
| P4 | 新产品 |
| P5 | 产品更新或调整 |
| P6 | Listing & Delisting |
| P7 | 其他公告，以及发布时间早于当前批次30天以上的历史补录 |

同一P级内按发布时间从新到旧排序，每个平台最多展示2条，最终取前5条。P级不读取高/中/低和差异标签。

### 12.3 业务使用原则

高/中/低回答“这条内容综合来看是否值得关注”；P1～P7回答“Overview 首页先展示哪一类内容”。高优先级不等于必须立项，P1也不一定比所有高优先级产品更重要。真正决定是否跟进时，仍需结合目标市场、战略相关性、投入成本和原文证据。

## 13. Action / Follow-up 当前如何处理

当前 Action 和 Follow-up 的业务解释力不足，因此暂不参与页面展示、列表列项、详情摘要或 Overview 重点摘要。页面继续展示公告事实、AI Summary、分析优先级和 Zoomex Comparison，具体行动由运营人员结合这些证据判断。

历史数据中的相关字段暂时保留，避免破坏既有数据兼容性；保留不代表其结论已被业务采纳。

## 14. 每日结果如何进入飞书

日常流程完成后有两个飞书出口，业务上应这样理解：

### 14.1 三张多维表

系统把当天采集到的新增或更新内容分别同步到三张业务表：

- Campaign；
- Product；
- Listing & Delisting。

三张表不是数据库的替代品，而是方便业务协作、筛选和补充备注的业务视图。Campaign/Product 使用与看板相同的差异标签；Listing & Delisting 不写入 Zoomex 对比结论。

### 14.2 EN 群日报

群推送不再按多个 Tab 和多个地区群分别发送。一份 EN 群日报只产生一条交互卡片消息：上方以两个独立视觉模块展示 Campaign Summary 和 Product Summary，不再重复推送 Overview 综合 Summary；中间展示 Overview 截图；最下方依次提供可视化看板主入口，以及 Campaign、Product、Listing & Delisting 三张飞书表入口。系统截图前会显式点击顶部“最新批次”。

整张卡片由应用机器人发送，不使用业务群 webhook。若截图生成或上传临时失败，系统会降级发送不含图片的文本卡片，确保当天摘要和业务入口仍可到达。

因此，群里的截图代表最近一次成功批次的 Overview 快照，不代表接收人当前在浏览器中选择的筛选条件。卡片中的摘要来自每日流程已经生成的 LLM 缓存，推送时不会再次调用模型。需要核查明细时应使用卡片按钮打开看板或对应飞书业务表。

## 15. 数据标记与可信度说明

### 15.1 “模拟”标记

如果分析结果来自演示或模拟数据，页面会显示“模拟”。这类内容只用于验证看板展示，不应作为正式业务结论。

### 15.2 “由 EN 派生”

同一业务事件的多个语言版本可能复用 EN 的机制分析，以减少重复分析。

这意味着玩法和差异判断来自同组英文版本，不意味着本地语言原文不存在。若不同市场版本实际包含特殊规则，仍需打开当地语言原文复核。

### 15.3 老数据降级

历史批次可能缺少后来新增的字段。系统会显示空值或待分析，不会用猜测值填充，也不会因为缺字段导致整页无法展示。对于已经完成 Campaign/Product 分析但旧标签仍为空的记录，当前导出规则会统一归入“未检索到同类”；真正从未分析的记录才保留“未进行对比”。

### 15.4 数据覆盖限制

使用看板时应了解：

- Lbank 等源站可公开获取的列表窗口有限，历史可能不完整；
- 源站没有写出的奖励、时间或目标用户，系统无法可靠补全；
- 分类和结构化提取可能因公告表述模糊而出现误差；
- 地区独占表示公告覆盖差异，不必然表示功能权限差异；
- ZMX 缺失结论依赖当前已采集的 ZMX 历史完整性和分类目录；
- ZMX 能力目录会随着新公告重新汇总，目录描述可能变化；
- 逐条差异结论反映当次分析，不会因为目录后来更新而自动重写。

## 16. 推荐的业务使用方式

### 每日快速浏览

1. 先看 Overview 的四类变化量；
2. 阅读 Highlights，优先关注 P1～P5；
3. 查看 Today’s Insight 是否为真实 LLM 结果；
4. 对“ZMX 缺失”或“玩法不同”的项目展开详情；
5. 点击竞品和 ZMX 原文确认关键信息。

### 活动运营分析

1. 进入 Campaign；
2. 按市场和活动类型筛选；
3. 对比奖励、门槛、目标用户和活动周期；
4. 查看相同机制是否被多家竞品采用；
5. 结合多家竞品采用情况，由运营人员决定是否进入调研。

### 产品竞品分析

1. 进入 Product；
2. 区分 New Product 和 Product Update；
3. 按产品类别过滤；
4. 查看 ZMX 是缺失、已有还是机制不同；
5. 结合 ZMX 对照公告确认差异是在能力、规则还是包装表达层面。

### 区域运营分析

1. 进入 Markets；
2. 选择目标市场；
3. 查看区域独占事件；
4. 对照其他语言版本确认是否只是公告发布时间差异；
5. 核实当地资格、奖励和合规描述。

## 17. 常见误读

| 页面现象 | 不应这样理解 | 正确理解 |
|---|---|---|
| 状态为“新增” | 竞品今天刚发布 | 系统本批次第一次采集到，需同时看发布时间 |
| 没有 ZMX 对照公告 | ZMX 一定没有 | 当前没有具体证据实例，还要看正式差异结论和分析状态 |
| 待分析 | ZMX 与竞品无差异 | 尚未完成比较，不能下结论 |
| 高优先级 | 必须立即跟进 | 值得优先人工评估，不等于自动立项 |
| P1 | 一定比所有其他内容重要 | 只是 Overview 固定展示顺序，不是综合评分 |
| 已有同类型 · 粗粒度 | Zoomex 的具体玩法完全相同 | 只确认做过同一大类，未完成精确逐条匹配 |
| 地区独占 | 功能只在该地区开放 | 当前公告只覆盖部分地区，权限需看正文 |
| 趋势当天升高 | 竞品当天集中发文 | 当天系统发现的新增/变化增加，可能包含补录 |
| Listing 没有优先级 | 数据漏分析 | 当前政策只做事实和赛道分类，不做综合优先级 |
| 群里只有 Overview | 其他 Tab 没有生成 | 群策略就是只发送最新批次 Overview，详情仍在看板和三张业务表中 |

## 18. 日常完整流程与结果检查

一次正常的每日执行会依次完成：

1. 采集指定 UTC 当天发现的新增或更新公告；
2. 清理重复内容、确认跨语言归组和业务分类；
3. 更新 Zoomex 能力目录；
4. 分析 Campaign/Product，并完成差异和高/中/低优先级计算；
5. 分类 Listing & Delisting 的币种赛道；
6. 生成 Overview、Campaign、Product 三份2～4句 Insight；
7. 导出看板数据并检查关键字段是否完整；
8. 将当天结果同步到 Campaign、Product、Listing & Delisting 三张飞书表；
9. 将 Campaign、Product 两份 Summary、Overview 截图与四个业务入口组成交互卡片，通过应用机器人发送到 EN 群；
10. 打开看板、选择顶部“最新批次”，截取 Overview 并发送至 EN 群。

如果关键分析字段、三份 Summary 或看板数据验收失败，流程会停止，不应把残缺结果当作正式日报继续发送。

业务人员可用以下方式快速验收：

- Overview 四张卡的总量与各详情 Tab 数量在相同筛选条件下能够对应；
- Campaign/Product 已分析数据中没有“未进行对比”；
- Listing & Delisting 没有 Zoomex 对比和高/中/低优先级；
- 三份 Insight 的描述与当前最新批次数据一致；
- 飞书三张表分别只包含自己的业务类别；
- EN 群截图顶部“最新批次”处于选中状态。

## 19. 简要技术附录

业务逻辑背后的实现采用“数据库快照 + 静态页面”方式：

- SQLite 保存公告、历史版本、分析结果和 ZMX 能力目录；
- Python 将当前数据库整理成 `docs/data/dashboard.json`；
- `docs/index.html` 读取 JSON，在浏览器中完成筛选、排序、展开和 CSV 下载；
- 页面本身不会修改数据库，也不会在打开时调用模型；
- 数据更新后需要重新导出 JSON 并发布页面。

主要文件：

| 内容 | 位置 |
|---|---|
| 看板页面 | `docs/index.html` |
| 看板数据快照 | `docs/data/dashboard.json` |
| 看板数据整理规则 | `src/dashboard/export_data.py` |
| Campaign/Product 分析 | `src/analysis/run.py` |
| ZMX 能力目录 | `src/analysis/zmx_catalog.py` |
| ZMX 标准机制分类 | `config/zmx_mechanism_taxonomy.yaml` |
| 分类规则 | `config/category_mapping.yaml` |
| 每日完整流程 | `scripts/run_daily.sh` |

历史数据需要补跑 Listing/Delisting Category 时执行：

```bash
python -m scripts.backfill_listing_categories \
  --db data/competitor_intel.db \
  --provider cursor_agent
```

该命令覆盖所有历史状态（包括当前已经变成 unchanged 的公告），按 50 条分块分类并复用缓存；只有一个业务批次全部完成后才写入结果。日常新增和变化数据不需要单独执行此命令，`scripts/run_daily.sh` 已封装采集、分析、三份 Summary、看板、飞书三表、日报卡片和 Overview 群截图的完整流程。
